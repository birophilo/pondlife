## What `markRaw` means (Vue 3)

`markRaw(someObject)` tells Vue: **do not wrap this object in a reactive Proxy**.

- If you put that object inside `reactive()`, `ref()`, or Pinia state, Vue **skips** making it (and typically **does not** deep-track its properties — the reference is stored as-is).
- **Mutating** `someObject.position.x` after that **does not** go through Vue’s `set` / `trigger` — so you **don’t** pay reactivity cost on every tick, and **nothing** re-renders automatically when those fields change.

So in your Performance trace, the expensive `set` → `trigger` → `job` chain is exactly what **`markRaw` is meant to avoid** for hot-path objects: they stop being “reactive.”

**Tradeoff:** anything that depended on “agent object is reactive” for **automatic** UI updates **stops** updating unless you wire something else.

---

## Keeping agents “outside” Pinia / less reactive

Rough patterns (often combined):

1. **`markRaw(agent)`** when creating / before pushing into `store.agentItems`  
   - Agents stay in Pinia for **lookup and non–hot-path** use, but **per-frame mutations** don’t notify Vue.

2. **Plain module / class registry**  
   - e.g. `const world = new Map()` or `agentsById` **outside** Pinia, filled on load/spawn.  
   - Pinia only holds **ids**, scene metadata, **selected id**, UI flags.

3. **`shallowRef` / `shallowReactive`** for containers  
   - Replacing the **array** or **map** notifies; **deep** property changes on nested objects may not — often used with **explicit** “version” bumps. Less ideal than `markRaw` for “many agents mutating every frame” if you’re not careful.

The theme: **simulation truth** = plain JS; **Vue** = thin layer for what the UI actually needs.

---

## How the UI (e.g. selected agent X/Y) still updates

If agents are `markRaw` / plain, **nothing** in the template can “subscribe” to `position.x` directly. You bridge **only what the UI needs**, at a **controlled rate**.

### Option A — **Single “inspector snapshot” ref (very common)**

- Keep a **`ref` or small `reactive` object** used **only** by the inspector, e.g. `selectedAgentSnapshot`.
- In **`animate()`** (or right after your sim step), **if** there is a selected agent:
  - `selectedAgentSnapshot.value = { x, y, ... }` (copy from the **plain** agent).
- That’s **at most one reactive write per frame** (and only when a selection exists), not **per agent**.

You can throttle to **every 2–4 frames** or **~100 ms** if the panel doesn’t need pixel-perfect 60 Hz text.

### Option B — **Pinia: only selection + snapshot**

- Store: `selectedAgentId` + optional `inspectorFields` updated from the loop as above.  
- Agents themselves can live in a non-reactive world; Pinia is **coordination + UI-facing slice**.

### Option C — **Imperative DOM** (niche)

- Update a **plain DOM node** or **tiny non-Vue widget** from `animate` for high-frequency numbers — bypasses Vue entirely for that strip of UI.

---

## “App-wide update per agent per tick?” vs “per tick only”

Today, if every agent is a **reactive** object, Vue can be doing work proportional to **writes × subscribers** (and internal bookkeeping). That’s much worse than “one panel update.”

With **`markRaw` + snapshot**:

- **Sim loop:** **zero** Vue notifications from moving every agent (ideal).
- **Inspector:** **one** reactive update **per frame** (or per throttle interval) **only when** you care (e.g. something is selected and you want live coords).

So yes: **narrow the reactive surface** to **“per tick, only for UI that must move”** — often **one** selected agent or **one** small struct, not **every** agent.

---

## Summary

| Mechanism | Role |
|-----------|------|
| **`markRaw` / plain world** | Stop per-frame agent mutations from driving Vue’s reactivity. |
| **Snapshot `ref` / small store slice** | Drive **selected agent** (or metrics) panel at **1× per frame** or **throttled**. |
| **Throttle** | Trade “perfectly live” for fewer re-renders if labels don’t need 60 Hz. |

So: **`markRaw` isn’t “hide data from the page”** — it’s **“stop Vue from treating this as reactive.”** You **explicitly** copy the few fields the UI needs into something reactive **when** and **how often** you choose — which is exactly how you avoid both **global per-agent reactivity** and **stale** inspector UI.

## What many production setups do (the common thread)

There isn’t one universal stack, but there **is** a repeated architectural rule:

**The `requestAnimationFrame` loop should not drive the framework’s reactive/render pipeline at full frame rate.**

So the “standard” idea is **separation of concerns**:

| Layer | Responsibility |
|--------|----------------|
| **Game / sim** | Plain JS (or engine) state, rAF loop, canvas drawing, input → **no** React/Vue state updates per frame. |
| **Chrome / site** | Router, layout, modals, settings, auth — **event-driven** or **low-rate** updates (e.g. 4–10 Hz, or only on user action). |

How that shows up in the wild:

- **Browser games** (Phaser, Pixi, custom WebGL): the **game** often lives in a **canvas + script** with its own loop; the **website** around it is separate HTML or a **minimal** framework shell (menus, shop, profile).
- **“React + game”** tutorials that perform well usually either: put the game in an **iframe**, use a **ref + imperative** canvas with **zero** `setState` in rAF, or use **engine-specific** integrations that **batch** and avoid full-tree React updates (still easy to get wrong).

- **Web Worker** (or WASM) for simulation; main thread **only draws** and passes buffers.
- Helps **CPU** if sim math dominates; **does not** fix Vue if the main thread still **re-renders a huge tree** every time something reactive flips.

### 4. **Reduce framework surface area**

- **Vue only for shell** (layout, router, modals); **canvas + loop in vanilla** (or a tiny custom layer).
- Or **another** UI layer for “tools” (Svelte/Solid) **only** if you’re doing a broader migration — not a small patch.

### 5. **Things to verify once**

- **Production build** (`npm run build` + serve) vs dev — Vue dev overhead alone can be **large**.
- **Performance** profile again: is time still in **`patch` / `render`** or mostly in **`drawImage` / your handlers**? That tells you whether the next step is **splitting Vue** or **optimizing sim/canvas**.
