Here’s a compact take aligned with your chat-history notes and the router regression.

---

## Why Vue Router might tank FPS (even with one route)

Your earlier diagnosis was: **reactive work every frame** (store / vdom), not “canvas drawing is slow.” Vue Router adds:

- Another **component layer** (`App` → `router-view` → `SimView`).
- **Reactive route state** (`route`, matched records) anywhere something uses `useRoute()` / `useRouter()` or the template touches `$route`.
- **Async route components** (`() => import(...)`) — extra chunk loading and a slightly different mount story (usually one-time, not per frame).

A jump **60 → 6 FPS** usually means something is again causing **large, frequent updates** (whole `SimView` re-rendering or heavy dependency tracking), not the router “doing routing” once. It’s worth **profiling** (Vue DevTools → highlight updates, or Performance tab) to see *what* re-renders each frame.

**Other obvious hot-path isolations** (beyond hover + `animationFrameId`):

1. **Nothing in `requestAnimationFrame` / `animate()` should assign to reactive state** used in templates — including `ref`/`reactive` in `SimView`, Pinia, or props. Keep using module-level vars + `getGlobals()` (and imperative DOM like cursor) for per-frame data.
2. **Don’t read `route` in the template** of the sim shell if you can avoid it; avoid `useRoute()` in anything that re-renders with the canvas loop. If you need the path, read it once or subscribe narrowly.
3. **`markRaw`** for big sim objects if they ever sit in reactive trees; **`shallowRef`** if a big blob must live in Vue.
4. **Throttle** live metrics / FPS text (you already discussed updating refs at ~2 Hz, not 60 Hz).
5. **Split UI**: a **minimal** parent that only hosts `<canvas>` + rAF, and **slower** panels as siblings with isolated state — so form/store churn doesn’t invalidate the canvas host every frame.

---

## Would a “hand-rolled router” be faster?

**Maybe slightly**, in the sense of **no** `vue-router` package, **no** reactive `route` object, and **no** matched-route machinery — you could do `history.pushState` + `popstate` and `v-if`/`v-show` between a few top-level views.

But: **if the cost is “SimView + whole subtree re-rendering,”** swapping Vue Router for 20 lines of history API **won’t fix** that unless you also **stop** whatever is making that subtree reactive to the hot path. So hand-rolled routing is a **small** lever; **architecture of reactivity** is the big one.

---

## If you drop client-side routing

| Approach | Idea |
|--------|------|
| **Single URL + tabs** | No router: state for “which panel” is local/`v-if`, or hash-only (`#settings`) without Vue Router — **no** extra library, minimal overhead. |
| **Django (or any) MVC, full page loads** | “Routing” = real URLs, each request returns HTML. Fine for **separate** admin/setup vs game, **bad** for keeping a **running** canvas sim across navigations unless you accept reloads or duplicate apps. |
| **HTMX** | Server returns HTML **fragments**; `hx-push-url` / boosted links give **URL changes** without a JS router. Great for **form-heavy, document-style** UIs. For a **60 FPS canvas**, you’d typically keep the canvas in a **stable** page and use HTMX only for **surrounding** chrome or separate pages — not as a replacement for a game loop. HTMX doesn’t give you a **client-side** route map like Vue Router; it’s a different model (hypermedia). |

---

## Practical conclusion

- **Vue Router isn’t inherently “6 FPS,”** but it can **amplify** problems if the sim view is still tied to broad reactivity. Your chat logs already point to the right rule: **rAF must not feed the reactive graph every frame.**
- **Try without router first**: same `SimView` as **root `App` again** — if FPS returns, the issue is **interaction of router + tree + reactivity**, not canvas math.
- **If you need multiple URLs later**: prefer **minimal** routing (hash + `v-if`, or hand-rolled `pushState`) **plus** strict separation of the **canvas tick** from Vue state — or accept **full page** boundaries (Django pages) for rare navigations.

Here’s a **practical profiling strategy** aimed at “what’s updating too often?” in a Vue 3 + canvas app.

---

## 1. Before you measure

- **Use a dev build** first (Vue’s update tracking is clearer; DevTools hooks work). **Then** spot-check a **production build** — numbers will look better, but **relative** hotspots often stay the same.
- **Reproduce the bad case**: same scene, same agent count, router on/off if you’re comparing.
- **Close other heavy tabs** and DevTools **docked** vs **undocked** can change FPS; for *relative* comparison, keep layout consistent.

---

## 2. Vue DevTools — what to use

### A. **Component tree + “highlight updates”** (if your DevTools version has it)

- Open the **Components** panel, select **`SimView`** (or your root sim component).
- Turn on **highlight updates** / **flash on update** (wording varies by version).

**What to look for**

- **Green/flash every frame (~60/s)** on a large component → that component’s render is running every frame (bad if it’s the whole sim shell).
- **Flashing on `App` or `RouterView`** every frame → problem may be above the sim, or props/reactivity bubbling up.

**Interpretation**

- Canvas itself doesn’t “flash” as a DOM node — you’re seeing **Vue re-renders** of the **template** that contains the canvas and sidebars.

### B. **Timeline / Events** (Vue DevTools 6+)

- Record while the sim runs for a few seconds.

**What to look for**

- Bursts of **`component event`** / **render** / **update** tied to **`SimView`** or children **aligned with rAF** (every ~16 ms).
- **Pinia** or **store** mutations firing at high frequency (if the timeline shows store events).

---

## 3. Browser **Performance** panel (Chrome / Edge) — essential

This answers “*who* is expensive?” even when Vue DevTools is vague.

1. Open **Performance**, click **Record**, let the sim run **3–5 s**, stop.
2. In the timeline, find **Main** thread.

**What to look for**

- Long **yellow** blocks: **Scripting** — often your **`animate()`** + Vue scheduler + layout.
- Repeated patterns every **~16 ms**: note **function names** in the bottom-up / call tree (minified names in prod — use dev build or source maps).

**Useful subviews**

- **Bottom-Up** / **Call tree**: sort by **Self time** or **Total time**. Look for:
  - `patch` / `componentUpdateFn` / `renderComponentRoot` / `ReactiveEffect.run` → **Vue render pipeline**.
  - `flushJobs` / `queueJob` → **batch of reactive updates**.
- **Frame** strip: if **FPS** drops, see whether **one long task** per frame** or **many small tasks**.

---

## 4. What counts as “excessive updates” (for your app)

| Signal | Likely meaning |
|--------|----------------|
| **`SimView` (or root layout) re-renders ~60/s** | Something **reactive** used in that component’s template (or its children) **changes every frame** — e.g. store, `ref`, `computed`, or **`route`** if subscribed. |
| **Many child components re-rendering** | Props or store fields those children depend on change too often. |
| **Big scripting time, little Vue in stack** | Hot path is **your JS** (agents, canvas), not Vue — good; then optimize algorithms, not the framework. |
| **Vue `patch`/render high in stack every frame** | **Reactive → template** path is the problem — align with moving per-frame data **out** of reactive state. |

---

## 5. Targeted checks (quick, no full profile)

1. **Template grep** (mental or actual): does **`SimView`’s template** read anything that **changes every frame**? (`animationFrameId`, hover in store, `route.fullPath`, computed from agents…)
2. **Temporarily** comment out **suspicious** template sections (e.g. whole sidebar) — if FPS jumps, that block’s dependencies are involved.
3. **`useRoute()`**: if used in `SimView`, ensure **no** computed/template depends on `route` in a way that updates spuriously (rare but worth checking).

---

## 6. Optional: **Performance.mark** in `animate()**

Add **one** `performance.mark`/`measure` around your **full rAF callback** and around **only** `renderAgents` / canvas draw. Compare **time in rAF** vs **frame gap** — if frame budget is blown by **non-canvas** work, the Performance panel will already show Vue in the stack; marks just confirm.

---

## 7. What “success” looks like

- **Canvas / sim logic** uses CPU every frame (expected).
- **Vue render** (`patch`, component render functions) should **not** run at 60 Hz for the **entire** `SimView` unless you **intentionally** drive that much UI from the loop (usually you don’t).

---

**Short checklist**

1. DevTools **highlight updates** on `SimView` → flashing every frame?  
2. **Performance** recording → **Bottom-Up** → Vue `patch` / `render` every frame?  
3. Correlate with **what** in template/store/route changed (grep + temporary removal).

That’s enough to separate **“Vue is re-rendering too much”** from **“simulation/canvas math is too heavy”** and to decide whether dropping the router, splitting components, or moving state fixes the right layer.
